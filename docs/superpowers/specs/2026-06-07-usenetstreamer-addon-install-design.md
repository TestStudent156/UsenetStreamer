# UsenetStreamer: install & manage via Home Assistant add-on — Design

**Date:** 2026-06-07
**Status:** Approved (direction + key decisions); ready for implementation plan.

## Goal

Change the `usenetstreamer` integration so that Home Assistant **provisions and runs**
UsenetStreamer itself, instead of only monitoring an externally-running instance. HA does
this the only supported way a container can be installed from an integration: by installing
and managing a **Home Assistant add-on** via the Supervisor `AddonManager` (the Z-Wave JS /
Matter pattern).

## Why the literal "integration installs a container" is impossible without this

- A HA integration runs sandboxed inside HA Core; it has no Docker access and cannot start
  containers.
- The only HA mechanism to install/run a container is the Supervisor add-on system, which
  only works for things packaged as HA add-ons.
- UsenetStreamer is a standalone Node.js Docker app (`ghcr.io/sanket9225/usenetstreamer`),
  not a HA add-on, and no add-on package for it exists anywhere.

Therefore the project has **two deliverables**: (1) a HA add-on that wraps the upstream
image, and (2) an integration rework that installs/manages that add-on.

## Key facts established by research (2026-06-07)

- **Upstream image:** `ghcr.io/sanket9225/usenetstreamer`, published & multi-arch
  (`linux/amd64`, `linux/arm64`; **no armv7**). Pin tag `1.7.12`.
- **Base of upstream image:** `node:22-alpine`, `WORKDIR /usr/src/app`, `CMD ["npm","start"]`,
  `EXPOSE 7000`.
- **Required env:** only `ADDON_SHARED_SECRET` (app returns HTTP 503 until set; it is frozen
  and cannot be changed through the admin API). Optional: `ADDON_STREAM_TOKEN`,
  `ADDON_BASE_URL`, `PORT` (default 7000), `CONFIG_DIR` (persistent config dir; use
  `/data/config`).
- **Persistence:** `/data` is the always-mounted add-on volume; set `CONFIG_DIR=/data/config`
  so `runtime-env.json` (dashboard-saved settings) survives restarts.
- **Admin API (unchanged contract):** `GET`/`POST /admin/api/config` with header
  `X-Addon-Token: <ADDON_SHARED_SECRET>`. **This is exactly what the current `api.py`
  already speaks** — so `api.py`, `coordinator.py`, `entity.py`, `sensor.py`,
  `binary_sensor.py` need no changes. There is no dedicated health endpoint; a successful
  `GET /admin/api/config` (or a TCP check on 7000) is the readiness signal.
- **Target host:** HA OS 17.3, Supervisor 2026.05.1, Core 2026.6.1, **arch amd64**. Samba
  (`core_samba`) and Advanced SSH (`a0d7b954_ssh`) add-ons installed.
- **Local add-on:** folder dropped under the host `addons` share → slug
  `local_usenetstreamer`. Supervisor builds it from its Dockerfile on-device. Bump
  `version:` to force a rebuild after code changes.
- **AddonManager API (current core):** import `AddonError, AddonInfo, AddonManager,
  AddonState` from `homeassistant.components.hassio`; detect Supervisor with
  `homeassistant.helpers.hassio.is_hassio`. Each integration defines its own `@singleton`
  `get_addon_manager`. Methods: `async_get_addon_info`, `async_install_addon`,
  `async_set_addon_options`, `async_start_addon`, `async_stop_addon`,
  `async_uninstall_addon`, `async_get_addon_discovery_info`, plus non-blocking
  `async_schedule_*` task variants. `AddonInfo` has `state`, `options`, `hostname`,
  `version`, `update_available`, `available`.

## Decisions

1. **Add-on-install only.** The config flow always installs/uses the add-on. The previous
   manual host/port/admin_token flow is **removed**. The integration becomes
   **Supervisor-only and single-instance**.
2. **Verification delivery:** the add-on folder is copied to `\\192.168.0.139\addons` by us
   using the `core_samba` add-on credentials (collected at deploy time), then installed,
   configured, started, and probed via ha-mcp.
3. **Slug:** MVP targets the local add-on slug `local_usenetstreamer`. Publishing the add-on
   to a GitHub add-on repo (with the `store.add_repository` step + hash-prefixed slug) is a
   documented **follow-up**, not in this iteration.
4. **shared_secret:** auto-generated by the integration (`secrets.token_hex(16)`), shown to
   the user in the configure step (editable), set as the add-on option, and stored in
   `entry.data` as the `X-Addon-Token` value.

## Deliverable 1 — the add-on (`addons/usenetstreamer/`)

Files:

- **`config.yaml`**
  ```yaml
  name: UsenetStreamer
  version: "0.1.0"          # bump to force a local rebuild
  slug: usenetstreamer
  description: Run UsenetStreamer (Usenet -> Stremio) as a Home Assistant add-on.
  arch: [amd64, aarch64]
  startup: application
  boot: auto
  ports:
    "7000/tcp": 7000
  ports_description:
    "7000/tcp": UsenetStreamer web UI + admin API
  webui: "http://[HOST]:[PORT:7000]/"
  watchdog: "tcp://[HOST]:[PORT:7000]"
  panel_icon: mdi:download-network
  options:
    shared_secret: ""
    stream_token: ""
    base_url: ""
  schema:
    shared_secret: password
    stream_token: "password?"
    base_url: "url?"
  ```
  (`/data` is auto-mounted; no `map` needed. `image:` is intentionally omitted so the local
  add-on builds from the Dockerfile.)

- **`Dockerfile`** — thin wrapper (a pure `image:` add-on can't inject env vars):
  ```dockerfile
  FROM ghcr.io/sanket9225/usenetstreamer:1.7.12
  RUN apk add --no-cache jq
  COPY run.sh /run.sh
  RUN chmod +x /run.sh
  CMD ["/run.sh"]
  ```

- **`run.sh`** — POSIX sh + jq (no bashio dependency); reads `/data/options.json`, exports
  env, execs upstream:
  ```sh
  #!/usr/bin/env sh
  set -e
  OPTIONS=/data/options.json
  export ADDON_SHARED_SECRET="$(jq -r '.shared_secret // empty' "$OPTIONS")"
  STREAM_TOKEN="$(jq -r '.stream_token // empty' "$OPTIONS")"
  [ -n "$STREAM_TOKEN" ] && export ADDON_STREAM_TOKEN="$STREAM_TOKEN"
  BASE_URL="$(jq -r '.base_url // empty' "$OPTIONS")"
  [ -n "$BASE_URL" ] && export ADDON_BASE_URL="$BASE_URL"
  export CONFIG_DIR=/data/config
  mkdir -p "$CONFIG_DIR"
  cd /usr/src/app
  exec npm start
  ```

- **`README.md`**, **`icon.png`** (and `DOCS.md` optional).

## Deliverable 2 — integration rework

Unchanged: `api.py`, `coordinator.py`, `entity.py`, `sensor.py`, `binary_sensor.py`,
`services.yaml`, `apply_config` service handler.

- **`manifest.json`** — add `"after_dependencies": ["hassio"]` (soft dep).
- **`const.py`** — add `ADDON_SLUG = "local_usenetstreamer"`, `ADDON_NAME = "UsenetStreamer"`,
  `CONF_USE_ADDON`, `CONF_INTEGRATION_CREATED_ADDON`, default port `7000`. Keep existing
  `CONF_HOST/PORT/SSL/ADMIN_TOKEN/VERIFY_SSL` (reused to build the client from add-on info).
- **`addon.py`** (new) — `@singleton(DATA_ADDON_MANAGER)` `get_addon_manager(hass)` →
  `AddonManager(hass, LOGGER, ADDON_NAME, ADDON_SLUG)`; plus
  `async_ensure_addon_running(hass, entry)` helper (raises `ConfigEntryNotReady` while
  installing/starting).
- **`config_flow.py`** — single-instance, Supervisor-only:
  - `async_step_user`: if `not is_hassio(hass)` → `async_abort("not_hassio")`. Set unique id
    to `DOMAIN`, `_abort_if_unique_id_configured()`. Inspect add-on info; route to
    install/configure/start/finish based on `AddonState`.
  - `async_step_install_addon`: `async_show_progress` driving
    `async_schedule_install_addon`; on done → `configure_addon`; on error →
    `install_failed` → abort `addon_install_failed`. Set `integration_created_addon=True`
    when we installed it.
  - `async_step_configure_addon`: form pre-filled with a generated `shared_secret`
    (`secrets.token_hex(16)`); on submit, merge into add-on options + `async_set_addon_options`
    (only if changed) → `start_addon`.
  - `async_step_start_addon`: `async_show_progress` driving `async_schedule_start_addon`; on
    done → `finish_addon_setup`; on error → `start_failed` → abort `addon_start_failed`.
  - `async_step_finish_addon_setup`: read `async_get_addon_info()` for `hostname`; build
    connection (`host=hostname, port=7000, ssl=False, verify_ssl=True,
    admin_token=shared_secret`); validate via `client.async_validate()`; `async_create_entry`
    with `data` = connection + `use_addon`/`integration_created_addon`.
  - Options flow: keep `scan_interval` only (unchanged).
- **`__init__.py`** — `async_setup_entry`: if `entry.data[CONF_USE_ADDON]`, call
  `async_ensure_addon_running` first (raises `ConfigEntryNotReady` while booting). Build the
  client from `entry.data` (as today). Register `apply_config` (unchanged). `async_unload_entry`:
  if `use_addon` and `entry.disabled_by`, `async_stop_addon` (best-effort). `async_remove_entry`
  (new): if `integration_created_addon`, `async_uninstall_addon`.
- **`strings.json` + `translations/{en,de}.json`** — replace the old `config.step.user`
  credential fields with the new flow: `progress` strings (`install_addon`, `start_addon`),
  `configure_addon` step (field `shared_secret`), and `abort` reasons (`not_hassio`,
  `addon_info_failed`, `addon_install_failed`, `addon_start_failed`,
  `already_configured`). Keep `entity.*` and `services.*` blocks. Update en + de in lockstep
  (hassfest).

## Testing

- **`tests/test_config_flow.py`** — rewritten: mock `is_hassio=True`, mock `get_addon_manager`
  /`AddonManager` (`AsyncMock`) to return `AddonInfo` in each `AddonState`; assert the flow
  installs → configures (sets options with generated secret) → starts → creates an entry with
  the derived connection. Add `not_hassio` abort test. Add `already_configured` single-instance
  test.
- **`tests/test_init.py`** (new or extend) — `async_setup_entry` raises
  `ConfigEntryNotReady` when add-on `NOT_RUNNING`; schedules start; unload stops add-on only
  when `disabled_by`; `async_remove_entry` uninstalls only when `integration_created_addon`.
- Existing `test_api.py`, `test_coordinator.py`, `test_sensor.py`, `test_services.py` remain
  (API/coordinator/entities unchanged). `test_services._setup` may need the entry data shape
  updated to the add-on flow.
- Match CI locally: `pytest -v`, plus hassfest shape (manifest + strings/services lockstep).

## Verification plan (live, via ha-mcp)

1. Collect `core_samba` credentials; copy `addons/usenetstreamer/` to
   `\\192.168.0.139\addons\usenetstreamer\`.
2. Supervisor store reload; confirm `local_usenetstreamer` appears (`ha_get_addon`).
3. Install, set `shared_secret` option (`ha_manage_addon` config mode), start (`hassio`
   service / supervisor API).
4. Confirm `state: started`; probe `GET /admin/api/config` with the token via
   `ha_manage_addon` proxy mode → expect a `values` payload.
5. Read add-on logs on any failure (`ha_get_logs source=supervisor slug=local_usenetstreamer`).

**Integration** new flow is interactive (config-flow wizard) and cannot be clicked through
via MCP; it is verified by the pytest suite, plus an optional manual UI walk by the user.

## Out of scope / follow-ups

- Publishing the add-on to a GitHub add-on repo + `store.add_repository` + hash slug
  (distribution).
- Ingress/TLS for the Stremio side (the app serves plain HTTP; Stremio needs HTTPS via an
  external proxy — unrelated to HA monitoring/config).
- Surfacing the ~98 optional config keys as add-on options (left to the in-app dashboard /
  `apply_config`).
